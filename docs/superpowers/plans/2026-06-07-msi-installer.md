# MSI Installer Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a per-machine Windows `.msi` installer for DeskTag (install to Program Files, Start-menu shortcut, autostart at logon, clean uninstall), built and published by GitHub Actions on tag push.

**Architecture:** `cargo-wix` turns a committed, hand-customized `wix/main.wxs` (WiX Toolset v3) into the `.msi`. A `.github/workflows/release.yml` on `windows-latest` builds the binary, runs `cargo wix`, and attaches `desktag.exe` + `.msi` to the GitHub Release for the pushed tag.

**Tech Stack:** Rust (existing), WiX Toolset v3, `cargo-wix`, GitHub Actions (`softprops/action-gh-release`).

**Note on testing:** This feature produces an installer + CI config, not Rust code, so there are no `cargo test` units. "Verification" steps are: (a) `cargo wix` compiles the `.msi` without error, (b) a manual install/uninstall checklist, (c) a green CI run producing the release assets. The existing `cargo test` suite (47 tests) must still pass on every commit because the pre-commit hook runs it.

## File Structure

- **`wix/main.wxs`** (new) — WiX v3 source. Owns the entire install definition: install dir, the `desktag.exe` component, autostart registry value, Start-menu shortcut, ARP icon, close-running-daemon action, launch-after-install action.
- **`.github/workflows/release.yml`** (new) — CI: build + package + publish on `v*` tag.
- **`Cargo.toml`** (modify) — version `1.0.0` → `1.0.1` (cargo-wix reads MSI `ProductVersion` from here).

`/target` and build artifacts stay git-ignored.

## Task 3: Customize `wix/main.wxs` — autostart, shortcut, icon, close, launch

**Files:**
- Modify: `wix/main.wxs`

All edits are in this one file. After editing, Step 7 verifies it compiles.

- [ ] **Step 1: Add the WixUtilExtension namespace**

On the root `<Wix ...>` element, add the `util` namespace so `util:CloseApplication` resolves:
```xml
<Wix xmlns="http://schemas.microsoft.com/wix/2006/wi"
     xmlns:util="http://schemas.microsoft.com/wix/UtilExtension">
```

- [ ] **Step 2: Add the product icon for Add/Remove Programs**

Inside `<Product>` (anywhere among its direct children), add:
```xml
<Icon Id="ProductICO" SourceFile="assets\desktag.ico"/>
<Property Id="ARPPRODUCTICON" Value="ProductICO"/>
```
If the generated file already has an `<Icon>`/`ARPPRODUCTICON` pointing at a `wix\` icon, repoint `SourceFile` to `assets\desktag.ico` and reuse its `Id` instead of adding duplicates.

- [ ] **Step 3: Add the autostart registry value**

Inside the main `<Component>` that contains the `desktag.exe` `<File>` (the one whose file id is `<EXE_FILE_ID>` from Task 2), add a sibling `<RegistryValue>`:
```xml
<RegistryValue Root="HKLM"
               Key="Software\Microsoft\Windows\CurrentVersion\Run"
               Name="DeskTag"
               Type="string"
               Value="&quot;[#<EXE_FILE_ID>]&quot;"
               KeyPath="no"/>
```
Replace `<EXE_FILE_ID>` with the actual `<File>` id. `[#id]` resolves to the installed exe's full path; the `&quot;` wrap it in quotes so a spaced path (`Program Files`) survives. Uninstall removes this value automatically because it belongs to the component.

- [ ] **Step 4: Add the Start-menu shortcut component**

Add a new top-level directory + component. Place this `<DirectoryRef>` as a child of `<Product>` (next to the existing `<Directory>` tree):
```xml
<DirectoryRef Id="ProgramMenuFolder">
  <Component Id="ApplicationShortcut" Guid="*">
    <Shortcut Id="AppStartMenuShortcut"
              Name="DeskTag"
              Description="Show the current virtual desktop name"
              Target="[#<EXE_FILE_ID>]"
              WorkingDirectory="APPLICATIONFOLDER"
              Icon="ProductICO"/>
    <RemoveFolder Id="CleanupShortcut" On="uninstall"/>
    <RegistryValue Root="HKCU"
                   Key="Software\DeskTag"
                   Name="installed"
                   Type="integer"
                   Value="1"
                   KeyPath="yes"/>
  </Component>
</DirectoryRef>
```
Notes: `ProgramMenuFolder` is a WiX built-in (the Start menu); the shortcut lands directly in it. The `HKCU` `RegistryValue` is the component's required `KeyPath` (a per-user marker; standard WiX shortcut idiom). `APPLICATIONFOLDER` is the install dir id from the generated file — if the generated id differs (e.g. `INSTALLDIR`), use that id instead.

- [ ] **Step 5: Reference the shortcut component in the Feature**

Find the `<Feature>` element and add a `<ComponentRef>` so the shortcut actually installs:
```xml
<ComponentRef Id="ApplicationShortcut"/>
```

- [ ] **Step 6: Add close-on-install and launch-after-install actions**

Add these as direct children of `<Product>`.

Close the running daemon before files are touched (prevents a locked-file reboot on upgrade/uninstall):
```xml
<util:CloseApplication Id="CloseDeskTag"
                       Target="desktag.exe"
                       CloseMessage="yes"
                       RebootPrompt="no"
                       Description="Closing DeskTag before continuing..."/>
```

Launch the app after a fresh install, in the invoking user's context (not the elevated installer's), so the tray badge appears immediately:
```xml
<Property Id="WixShellExecTarget" Value="[#<EXE_FILE_ID>]"/>
<CustomAction Id="LaunchApplication"
              BinaryKey="WixCA"
              DllEntry="WixShellExec"
              Impersonate="yes"
              Execute="immediate"
              Return="asyncNoWait"/>
<InstallExecuteSequence>
  <Custom Action="LaunchApplication" After="InstallFinalize">NOT Installed</Custom>
</InstallExecuteSequence>
```
`Impersonate="yes"` runs the exe as the logged-in user; `NOT Installed` means "only on fresh install, not on uninstall". `WixCA` is the WiX v3 util custom-action binary. If an `<InstallExecuteSequence>` already exists in the file, merge the `<Custom>` line into it rather than adding a second element.

- [ ] **Step 7: Verify the customized MSI compiles**

Run:
```powershell
cargo wix --nocapture --ext WixUtilExtension
```
Expected: PASS — produces `target\wix\desktag-1.0.0-x86_64.msi`. The `--ext WixUtilExtension` is REQUIRED now (both `util:CloseApplication` and `WixShellExec` live in that extension); without it WiX errors with unresolved `util:` / `WixCA`.

- [ ] **Step 8: Commit**

```powershell
git add wix/main.wxs
git commit -m "build: autostart, shortcut, icon, close+launch in installer"
```

## Task 5: GitHub Actions release workflow

**Files:**
- Create: `.github/workflows/release.yml`

- [ ] **Step 1: Write the workflow**

Create `.github/workflows/release.yml` with exactly:
```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

permissions:
  contents: write

jobs:
  build-msi:
    runs-on: windows-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Install Rust stable
        uses: dtolnay/rust-toolchain@stable

      - name: Install WiX Toolset v3
        shell: pwsh
        run: |
          choco install wixtoolset --no-progress -y
          $wixBin = (Get-ChildItem 'C:\Program Files (x86)\WiX Toolset*\bin' -Directory | Select-Object -First 1).FullName
          Write-Output "WiX bin: $wixBin"
          Add-Content -Path $env:GITHUB_PATH -Value $wixBin

      - name: Install cargo-wix
        run: cargo install cargo-wix

      - name: Build MSI (also builds the release binary)
        run: cargo wix --nocapture --ext WixUtilExtension

      - name: Collect artifacts
        shell: pwsh
        run: |
          New-Item -ItemType Directory -Force dist | Out-Null
          Copy-Item target\release\desktag.exe dist\
          Copy-Item target\wix\*.msi dist\

      - name: Publish release
        uses: softprops/action-gh-release@v2
        with:
          files: |
            dist/desktag.exe
            dist/*.msi
```
Notes: `cargo wix` builds the release binary itself, so no separate `cargo build` step. `softprops/action-gh-release@v2` creates the Release for the pushed tag (if absent) and uploads the assets using the built-in `GITHUB_TOKEN` granted by `permissions: contents: write`. `cargo install cargo-wix` is the slow step (~minutes) — acceptable; caching is out of scope.

- [ ] **Step 2: Sanity-check the YAML**

Run (if `actionlint` is available; otherwise visually confirm indentation):
```powershell

```
Expected: no errors. (Skip if `actionlint` is not installed — it is optional.)

- [ ] **Step 3: Commit**

```powershell

git commit -m "ci: build and publish MSI on tag push"
```
